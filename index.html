Modul: SQL Injection (SQLi)

Pendahuluan

Definisi bug: SQL Injection (SQLi) adalah kerentanan keamanan di mana penyerang memasukkan kode SQL berbahaya ke dalam input data yang tidak aman, yang kemudian dieksekusi oleh basis data. Dampaknya bisa berupa penguasaan basis data secara penuh.

Risiko dan dampak keamanan: Pencurian data sensitif, modifikasi atau penghapusan data, pengambilalihan akun administrator, hingga penolakan layanan (DoS).

Jenis-Jenis Bug

SQLi dibagi menjadi beberapa tipe, termasuk **In-band SQLi** (Error-based, Union-based) dan **Blind SQLi** (Boolean-based, Time-based).

Cara Manual Menemukan Bug

Step-by-step testing manual:

  1. Identifikasi input yang berinteraksi dengan database (form login, pencarian, parameter URL).
  2. Masukkan karakter ' (kutip tunggal) ke dalam input. Jika muncul pesan error SQL, itu adalah indikasi kuat.
  3. Uji kondisi boolean, misalnya dengan memasukkan ' OR 1=1-- di kolom username dan perhatikan respons halaman.

Payload Aman dan Efektif

Payload dasar:

' OR 1=1--

Payload bypass filter:

' or 1=1/*

Studi Kasus dan Proof of Concept (PoC)

Pada URL seperti http://site.com/profile?id=1, ubah menjadi http://site.com/profile?id=1' OR 1=1--. Jika halaman menampilkan data profil lain tanpa autentikasi, maka bug berhasil ditemukan.

Rekomendasi Mitigasi dan Perbaikan

Cara developer memperbaiki bug: Menggunakan Prepared Statements atau Parameterized Queries. Ini memisahkan kode SQL dari input pengguna.

Praktik coding aman: Validasi semua input pengguna dan gunakan prinsip least privilege pada database user.

Severity & CVSS

Klasifikasi tingkat risiko: SQLi hampir selalu High atau Critical.

Contoh scoring CVSS: 9.8 (Critical) dengan vektor AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:H/A:H.

Modul: Cross-Site Scripting (XSS)

Pendahuluan

Definisi bug: XSS adalah jenis kerentanan injeksi di mana penyerang menyuntikkan kode skrip berbahaya ke dalam halaman web yang dieksekusi oleh browser pengguna lain.

Risiko dan dampak keamanan: Pencurian data cookie dan sesi, pengalihan ke situs phishing, dan penyebaran malware.

Jenis-Jenis Bug

XSS terbagi menjadi tiga tipe utama: **Reflected**, **Stored**, dan **DOM-based**.

Cara Manual Menemukan Bug

Step-by-step testing manual:

  1. Identifikasi input yang menampilkan kembali data (komentar, pencarian, nama profil).
  2. Masukkan payload XSS dasar ke dalam input tersebut.
  3. Perhatikan apakah browser mengeksekusi kode tersebut (misalnya, muncul alert box).

Payload Aman dan Efektif

Payload dasar:

<script>alert('XSS')</script>

Payload bypass filter:

<img src=x onerror=alert(1)>

Studi Kasus dan Proof of Concept (PoC)

Pada sebuah halaman komentar, ketik <script>alert(document.cookie)</script>. Jika setelah di-submit muncul alert box berisi cookie, maka itu adalah bukti Stored XSS.

Rekomendasi Mitigasi dan Perbaikan

Cara developer memperbaiki bug: Terapkan sanitasi input dan **enkode output**. Sanitasi input menghapus karakter berbahaya, sedangkan enkode output memastikan data ditampilkan sebagai teks biasa.

Praktik coding aman: Gunakan framework dengan perlindungan XSS bawaan dan terapkan Content Security Policy (CSP).

Severity & CVSS

Klasifikasi tingkat risiko: XSS umumnya Medium hingga High.

Contoh scoring CVSS: 7.5 (High) dengan vektor AV:N/AC:L/PR:N/UI:R/S:C/C:L/I:L/A:N.

Modul: Local File Inclusion (LFI)

Pendahuluan

Definisi bug: File Inclusion adalah kerentanan yang memungkinkan penyerang menyertakan file, baik dari server lokal (LFI - Local File Inclusion) atau dari server eksternal (RFI - Remote File Inclusion).

Risiko dan dampak keamanan: Membaca file sensitif di server (misalnya file konfigurasi), hingga eksekusi kode server (RCE).

Jenis-Jenis Bug

Terdapat dua jenis utama: **Local File Inclusion (LFI)** yang menyertakan file dari sistem lokal, dan **Remote File Inclusion (RFI)** yang menyertakan file dari URL eksternal.

Cara Manual Menemukan Bug

Step-by-step testing manual:

  1. Cari parameter URL yang merujuk pada file, misalnya page.php?file=home.html.
  2. Coba ganti nilai parameter dengan file sensitif, seperti /etc/passwd atau ../../../../../../etc/passwd (untuk LFI).
  3. Untuk RFI, coba sertakan URL eksternal, seperti http://website.com/cmd.txt.

Payload Aman dan Efektif

Payload dasar LFI:

../../../../etc/passwd

Payload bypass filter:

....//....//....//....//etc/passwd

Studi Kasus dan Proof of Concept (PoC)

Pada URL http://site.com/index.php?file=konten.php, ubah menjadi http://site.com/index.php?file=../../../../etc/passwd. Jika isi file password server muncul, maka LFI berhasil.

Rekomendasi Mitigasi dan Perbaikan

Cara developer memperbaiki bug: Gunakan daftar file yang diizinkan (whitelist) dan hindari penggunaan input pengguna dalam fungsi `include()` atau `require()`. Jangan izinkan *remote file inclusion*.

Modul: Remote Code Execution (RCE)

Pendahuluan

Definisi bug: RCE adalah kerentanan yang memungkinkan penyerang mengeksekusi kode atau perintah sewenang-wenang di server dari jarak jauh.

Risiko dan dampak keamanan: RCE adalah salah satu bug paling parah. Dampaknya bisa berupa penguasaan penuh server, pencurian data, hingga penggunaan server untuk serangan lain.

Jenis-Jenis Bug

RCE dapat terjadi melalui **Command Injection**, **Code Injection**, atau kerentanan lain yang memungkinkan penyerang mengirimkan dan menjalankan kode di sistem target.

Cara Manual Menemukan Bug

Step-by-step testing manual:

  1. Cari fungsi yang mengeksekusi input pengguna, seperti exec(), system(), atau eval().
  2. Masukkan perintah dasar seperti whoami atau ls yang dibungkus dalam sintaks yang tepat.

Payload Aman dan Efektif

Payload dasar:

;id; (untuk bash)
;whoami (untuk bash)

Studi Kasus dan Proof of Concept (PoC)

Dalam form *command* di sebuah aplikasi, ketik ;ls -la. Jika daftar file di server muncul, maka RCE berhasil.

Rekomendasi Mitigasi dan Perbaikan

Cara developer memperbaiki bug: Hindari penggunaan fungsi yang mengeksekusi perintah shell. Jika harus digunakan, pastikan semua input pengguna divalidasi dan di-sanitasi dengan sangat ketat.

Severity & CVSS

Klasifikasi tingkat risiko: RCE selalu Critical.

Contoh scoring CVSS: 9.8 (Critical) dengan vektor AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:H/A:H.

Modul: Cross-Site Request Forgery (CSRF)

Pendahuluan

Definisi bug: CSRF adalah serangan yang memaksa pengguna terautentikasi untuk mengirimkan permintaan yang tidak disengaja ke server web yang rentan.

Risiko dan dampak keamanan: Mengubah kata sandi, melakukan transfer dana, atau mengirimkan data atas nama pengguna tanpa persetujuan.

Jenis-Jenis Bug

CSRF tidak memiliki jenis yang berbeda, tetapi bisa terjadi pada berbagai jenis permintaan HTTP (GET, POST, dll) dan memiliki tingkat dampak yang berbeda.

Cara Manual Menemukan Bug

Step-by-step testing manual:

  1. Identifikasi permintaan yang sensitif (misalnya, form ganti password).
  2. Tangkap permintaan tersebut menggunakan proxy seperti Burp Suite atau OWASP ZAP.
  3. Cek apakah permintaan tersebut berisi token anti-CSRF yang unik. Jika tidak ada, kemungkinan rentan.

Studi Kasus dan Proof of Concept (PoC)

Buat halaman HTML sederhana dengan form yang mengarah ke URL tindakan sensitif (misalnya, `gantipassword.php`). Jika pengguna yang sudah login mengunjungi halaman ini, permintaan akan terkirim secara otomatis.

Rekomendasi Mitigasi dan Perbaikan

Cara developer memperbaiki bug: Implementasikan token anti-CSRF unik untuk setiap sesi. Token ini harus diverifikasi di sisi server.

Praktik coding aman: Gunakan atribut SameSite=Lax atau SameSite=Strict pada cookie.

Severity & CVSS

Klasifikasi tingkat risiko: CSRF umumnya Medium hingga High. Dampaknya tergantung pada sensitivitas tindakan yang dapat dieksploitasi.

Contoh scoring CVSS: 8.8 (High) dengan vektor AV:N/AC:L/PR:N/UI:R/S:U/C:H/I:H/A:H.

Modul: Server-Side Request Forgery (SSRF)

Pendahuluan

Definisi bug: SSRF terjadi ketika aplikasi web tidak memvalidasi URL yang diberikan oleh pengguna, sehingga penyerang dapat memaksa server untuk membuat permintaan HTTP ke URL internal atau eksternal yang tidak semestinya.

Risiko dan dampak keamanan: Membocorkan informasi internal server, memindai jaringan internal, atau melakukan serangan Denial of Service (DoS).

Jenis-Jenis Bug

SSRF dibagi menjadi dua jenis utama: **Basic SSRF** (server merespons dengan hasil dari permintaan) dan **Blind SSRF** (server tidak memberikan respons langsung, sehingga memerlukan teknik out-of-band).

Cara Manual Menemukan Bug

Step-by-step testing manual:

  1. Identifikasi input yang menerima URL, misalnya parameter `url=` atau `image_url=`.
  2. Coba ganti URL dengan URL internal server, seperti http://localhost atau 127.0.0.1.
  3. Jika server merespons dengan isi file, itu adalah tanda SSRF.

Payload Aman dan Efektif

Payload dasar:

http://localhost
file:///etc/passwd

Modul: Insecure Direct Object Reference (IDOR)

Pendahuluan

Definisi bug: IDOR terjadi ketika aplikasi web menggunakan input pengguna untuk mengakses objek secara langsung, seperti ID file atau ID pengguna, tanpa melakukan validasi akses yang memadai.

Risiko dan dampak keamanan: Membaca, mengubah, atau menghapus data milik pengguna lain dengan hanya mengubah parameter URL atau form.

Cara Manual Menemukan Bug

Step-by-step testing manual:

  1. Identifikasi parameter yang merujuk pada objek, misalnya `profile?id=123`.
  2. Ubah ID tersebut menjadi ID lain yang mungkin, seperti `id=124` atau `id=1`.
  3. Jika kamu bisa melihat data pengguna lain, itu adalah IDOR.

Contoh POC

Pada URL untuk melihat invoice, /invoice?id=100, coba ubah menjadi /invoice?id=101. Jika invoice pengguna lain dapat dilihat, maka kerentanan IDOR terdeteksi.

Modul: Authentication Bypass

Pendahuluan

Definisi bug: Authentication Bypass adalah jenis kerentanan di mana penyerang dapat melewati mekanisme otentikasi suatu aplikasi web, seperti form login, untuk mendapatkan akses ke akun pengguna atau area terlarang.

Risiko dan dampak keamanan: Pengambilalihan akun pengguna, akses ke panel admin, atau akses ke data sensitif tanpa kredensial yang sah.

Jenis-Jenis Bug

Authentication Bypass bisa terjadi melalui berbagai cara, seperti **SQL Injection**, **Session Fixation**, atau manipulasi parameter URL.

Cara Manual Menemukan Bug

Step-by-step testing manual:

  1. Coba masukkan payload SQLi di form login, seperti ' OR 1=1--.
  2. Coba akses URL admin secara langsung, misalnya /admin.
  3. Periksa apakah aplikasi melakukan validasi sesi dengan benar.

Modul: File Upload Vulnerability

Pendahuluan

Definisi bug: Kerentanan File Upload terjadi ketika aplikasi web mengizinkan pengguna mengunggah file tanpa validasi yang memadai. Penyerang dapat mengunggah file berbahaya (misalnya, shell) untuk mendapatkan Remote Code Execution (RCE) di server.

Risiko dan dampak keamanan: RCE, deface situs web, atau pencurian data.

Cara Manual Menemukan Bug

Step-by-step testing manual:

  1. Identifikasi fitur unggah file.
  2. Coba unggah file dengan ekstensi berbahaya, seperti `.php`, `.jsp`, atau `.asp`.
  3. Coba unggah file dengan ekstensi ganda, misalnya `file.jpg.php`.
  4. Gunakan Burp Suite untuk memanipulasi Content-Type header dari `image/jpeg` menjadi `text/plain`.

Payload Aman dan Efektif

<?php system($_GET['cmd']); ?>

Modul: Command Injection

Pendahuluan

Definisi bug: Command Injection adalah kerentanan di mana penyerang dapat mengeksekusi perintah pada sistem operasi host melalui input yang tidak divalidasi, seperti form atau parameter URL.

Risiko dan dampak keamanan: Sama dengan RCE. Penyerang dapat mengontrol server sepenuhnya.

Cara Manual Menemukan Bug

Step-by-step testing manual:

  1. Cari parameter yang memproses perintah sistem, misalnya fitur ping.
  2. Masukkan payload dengan pemisah perintah, seperti `ip=127.0.0.1; ls`.
  3. Perhatikan output yang dihasilkan. Jika ada daftar file, berarti berhasil.

Payload Aman dan Efektif

Payload dasar:

; ls
| whoami
&& cat /etc/passwd

Modul: Open Redirect

Pendahuluan

Definisi bug: Open Redirect adalah kerentanan di mana aplikasi web mengizinkan penyerang untuk memanipulasi parameter URL yang digunakan untuk redirect pengguna ke URL eksternal yang berbahaya.

Risiko dan dampak keamanan: Digunakan untuk serangan phishing yang menipu pengguna agar berpikir mereka berada di situs yang sah.

Cara Manual Menemukan Bug

Step-by-step testing manual:

  1. Cari parameter URL seperti `redirect=`, `url=`, atau `next=`.
  2. Coba ganti nilainya dengan URL situs berbahaya, misalnya `http://google.com`.
  3. Jika kamu dialihkan ke URL tersebut, maka bug ini terdeteksi.

Payload Aman dan Efektif

Payload dasar:

https://www.google.com
//www.google.com

Catatan: Banyak situs menggunakan validasi yang ketat. Kamu bisa coba berbagai variasi untuk bypass filter.

Modul: Broken Access Control

Pendahuluan

Definisi bug: Broken Access Control adalah kerentanan di mana pengguna yang tidak sah dapat mengakses fungsionalitas atau data yang seharusnya terbatas untuk pengguna lain, seperti admin.

Risiko dan dampak keamanan: Pengungkapan data sensitif, modifikasi atau penghapusan data pengguna lain, hingga pengambilalihan akun administrator.

Jenis-Jenis Bug

Banyak jenisnya, termasuk **Horizontal Privilege Escalation** (mengakses data pengguna lain) dan **Vertical Privilege Escalation** (mendapatkan akses admin).

Cara Manual Menemukan Bug

Step-by-step testing manual:

  1. Login sebagai pengguna biasa (user A).
  2. Cari URL yang mengarah ke area sensitif, misalnya `dashboard-admin`.
  3. Coba akses URL tersebut. Jika berhasil, itu adalah Broken Access Control.

Studi Kasus dan Proof of Concept (PoC)

Login sebagai pengguna biasa. Perhatikan parameter URL, misalnya `dashboard?user_id=123`. Coba ubah menjadi `user_id=1` (ID admin). Jika halaman admin terbuka, itu adalah bukti Broken Access Control.
